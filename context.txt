4. DATABASE SCHEMA
============================================================

TABLES
------
classes                  — permanent class slots (id, name, level, day, time, end_time, venue, loc)
blocks                   — each 6-week block run (id, class_id, start_date, end_date, weeks, dates[], price, cap, booked, visible, status)
bookings                 — client bookings (id, class_id, block_id, customer_id, status, amount_due, refund_amount, created_at)
customers                — client records (id, first_name, last_name, email, phone, customer_type, priority, created_at)
                           NOTE: customers.priority column is retained in DB but no longer used by the app.
                           Per-class manual priority is now managed via customer_class_priority table.
parq                     — health forms (id, booking_id, customer_id, age, emergency_name, emergency_relationship, emergency_phone, q1_heart...q12_other_reasons, yes_details, additional_notes, print_name, sign_date)
                           NOTE: sign_date is DATE type. Front-end stores ISO (YYYY-MM-DD) in a data-iso
                           attribute on the display element and sends that direct to the DB.
                           Admin PAR-Q view uses formatDateDisplay() to render "18 Apr 2026".
settings                 — key/value store for bank details and admin email (bank_name, bank_sort_code, bank_account_no, admin_email)
waitlist                 — placeholder for future waitlist feature (id, class_id, customer_id, created_at)
                           RLS enabled. Empty — not yet connected to booking flow.
cancellations            — audit trail of all client removals from blocks
                           Columns: id, customer_id, class_id, block_id, first_name, last_name, email,
                           class_name, venue, block_start_date, block_end_date, sessions_attended,
                           sessions_remaining, price_per_session, refund_amount, refunded (BOOLEAN),
                           refunded_at (TIMESTAMPTZ), cancelled_at (TIMESTAMPTZ)
                           NOTE: first_name/last_name/email/class_name/venue are denormalised so records
                           survive even if the customer or class is later deleted.
customer_class_priority  — per-class manual priority grants
                           Columns: id (BIGSERIAL), customer_id (BIGINT FK → customers),
                           class_id (BIGINT FK → classes), granted_at (TIMESTAMPTZ)
                           UNIQUE constraint on (customer_id, class_id)
                           ON DELETE CASCADE on both FKs
                           A row present = that customer has manual priority for that class.
                           This table replaces the global customers.priority boolean.

NOTE: parq table uses emergency_name, emergency_relationship, emergency_phone (NOT emergency_contact — dropped)
NOTE: parq.booking_id FK is ON DELETE CASCADE — PAR-Qs auto-delete when their booking is deleted.
NOTE: bookings.block_id FK is ON DELETE CASCADE — prevents orphan bookings.
NOTE: App-level deletion code still clears child rows explicitly as belt-and-braces safety net.
NOTE: When bulk-deleting all client data, deletion order must be:
parq → cancellations → bookings → customer_class_priority → customers

CHECK CONSTRAINTS
-----------------
bookings:
  - bookings_status_check: status IN ('reserved','confirmed','cancelled','refund-pending','refunded','waitlist')

blocks:
  - blocks_status_check:    status IN ('upcoming','active','completed','cancelled')
  - blocks_price_nonneg:    price >= 0
  - blocks_cap_positive:    cap >= 1
  - blocks_weeks_positive:  weeks >= 1

customers:
  - customers_type_check:      customer_type IN ('new','returning','vip') OR customer_type IS NULL
  - customers_first_name_len:  char_length(first_name) <= 100
  - customers_last_name_len:   char_length(last_name)  <= 100
  - customers_email_len:       char_length(email)      <= 200
  - customers_phone_len:       char_length(phone)      <= 40

parq:
  - parq_yes_details_len:      char_length(yes_details)            <= 4000
  - parq_additional_notes_len: char_length(additional_notes)       <= 4000
  - parq_emergency_name_len:   char_length(emergency_name)         <= 100
  - parq_emergency_rel_len:    char_length(emergency_relationship) <= 50
  - parq_emergency_phone_len:  char_length(emergency_phone)        <= 40
  - parq_print_name_len:       char_length(print_name)             <= 100

IMPORTANT: customer_type must be lowercase 'new' or 'returning' — the
customers_type_check constraint rejects capitalised variants.

NOT NULL CONSTRAINTS
--------------------
customers:    email NOT NULL
bookings:     customer_id, class_id, block_id NOT NULL
blocks:       class_id, start_date, end_date NOT NULL
classes:      name, day, venue NOT NULL

NOT NULL violation = Postgres error code 23502; existing try/catch blocks
surface a generic toast ("Error saving class." / "Error adding block.").

PARTIAL UNIQUE INDEX
--------------------
bookings_unique_active_per_block:
  CREATE UNIQUE INDEX ON bookings (customer_id, block_id) WHERE status != 'cancelled'

Prevents duplicate non-cancelled bookings by same customer on same block, at DB level.
Re-booking after cancellation still works — cancelled rows excluded.
book_if_available RPC catches unique_violation and re-raises as ALREADY_BOOKED
so the front-end handles it identically.

SECURITY DEFINER FUNCTIONS
--------------------------
These bypass RLS and are called from the JS client instead of direct table access:

- lookup_customer(p_email) — returns customer record by email
- upsert_customer(p_first_name, p_last_name, p_email, p_phone, p_customer_type)
    inserts or updates customer, returns id
- book_if_available(p_block_id, p_class_id, p_customer_id, p_amount_due)
    atomic capacity check + booking insert, returns booking id
- check_priority_access(p_email, p_block_id) → BOOLEAN
    checks customer_class_priority (per-class manual) then previous-block
    confirmed booking
- has_active_booking_on_block(p_customer_id, p_block_id) → BOOLEAN
    true if customer has a non-cancelled booking on this block.
    Used by the booking modal step-1 handler.
- get_schema_fingerprint() → (fingerprint TEXT, total_objects BIGINT)
    Read-only deterministic hash of the public schema's structure (columns,
    constraints, indexes, RLS policies, functions, triggers, grants).
    Self-excluded from its own fingerprint. Used by the schema-check helper.

All functions granted EXECUTE to anon and authenticated roles.

DATABASE TRIGGERS
-----------------
trg_sync_block_booked_count — fires AFTER INSERT OR DELETE on bookings.
Calls sync_block_booked_count() which recalculates blocks.booked from actual
bookings (excluding cancelled). SECURITY DEFINER, SET search_path = public.

IMPORTANT: This trigger fires on app-level inserts/deletes only. If bookings
are bulk-modified via raw SQL, the booked count will NOT update automatically.
After raw SQL changes to bookings, run this resync:

    UPDATE blocks b
    SET booked = (
      SELECT COUNT(*) FROM bookings
      WHERE block_id = b.id AND status != 'cancelled'
    );

RLS POLICIES
------------
RLS enabled on all tables.

classes:                 SELECT anon+auth | INS/UPD/DEL auth only
blocks:                  SELECT anon+auth | INS/UPD/DEL auth only
bookings:                INSERT anon+auth (routes via SECURITY DEFINER) | SELECT/UPD/DEL auth only
customers:               INSERT anon+auth (routes via SECURITY DEFINER) | SELECT/UPD/DEL auth only
parq:                    INSERT anon+auth | SELECT/DEL auth only (no UPDATE — write-once)
settings:                SELECT anon+auth | INS/UPD auth only
waitlist:                ALL auth only
cancellations:           ALL auth only
customer_class_priority: ALL auth only

NOTE: customers and bookings have no anon SELECT policy — intentional. All
public reads of customer/booking data must go via SECURITY DEFINER functions.

ANON ROLE GRANTS
----------------
Table grants to anon:
  classes   — SELECT only
  blocks    — SELECT only
  settings  — SELECT only
  parq      — INSERT only (PAR-Q save during booking)

Tables with NO anon grants at all:
  bookings, customers, cancellations, waitlist, customer_class_priority

Function EXECUTE grants to anon:
  book_if_available, check_priority_access, has_active_booking_on_block,
  lookup_customer, upsert_customer, get_schema_fingerprint

The `authenticated` role retains ALL on every table (admin dashboard unaffected).

TEST / PROD SCHEMA PARITY

Both projects have a SECURITY DEFINER function get_schema_fingerprint()

constraints, indexes, RLS policies, functions, triggers, and grants in
the public schema. Read-only, granted to anon, exposes nothing sensitive.

and compares the results. Auto-generated constraint and index names are

DATABASE & SUPABASE
-------------------
- customer_type values must be lowercase: 'new', 'returning', 'vip' (or NULL).
- blocks.dates[] stores display-formatted strings ("6 Apr", "13 Apr"), NOT ISO dates.
- Front-end JS must never rely on direct SELECT against customers or bookings
  from an anon context — RLS silently returns empty arrays rather than erroring.
- Three-layer defence against duplicate bookings (DB unique index, RPC catch,
  front-end early detection).
- PAR-Q presence is flagged in the dashboard for new-client bookings only.
- Supabase web SQL editor has a parser bug with PL/pgSQL DECLARE vars.
  Use apply_migration (MCP) or Supabase CLI for any DDL involving function bodies.
- supabase_migrations.schema_migrations stores full SQL of applied migrations
  in its `statements` column — useful for reconstructing migrations applied
  via MCP without a local file.
- Test admin users on Supabase MUST be created via the dashboard, not via raw
  SQL. Raw inserts into auth.users don't populate auth.identities, which
  Supabase Auth requires for password login.
- After raw SQL changes to bookings, manually resync blocks.booked — the
  trigger only fires on app-level inserts/deletes (see DATABASE TRIGGERS).
- The lg-pilates-test Supabase project is a byte-for-byte mirror of production
  schema. Any future production schema changes MUST be mirrored to test or
  tests will drift from reality.

- Class card root selector is `.card`, NOT `.class-card`. Several Session 18

  fix (Option B in TO DO) is to add afterEach cleanup to every spec that
  creates state. Until that lands, expect to occasionally need a manual
  cleanup or full `npm run seed` after heavy iteration.

SESSION 20 LEARNINGS — PARQ 401 ROOT CAUSE, .SELECT() CHAINING PITFALL
---------------------------------------------------------------------
- Root cause of the CB-33 parq 401: the front-end's parq insert was
  `await sb.from("parq").insert({...}).select()`. The trailing `.select()`
  makes PostgREST add `RETURNING *`, which requires SELECT permission.
  Anon has INSERT but not SELECT on parq, so PostgREST returned 42501.
  The booking flow's try/catch swallowed the error and showed the
  success view — so the bug was invisible in production. The CB-33 test
  was the only thing that surfaced it.
- The lingering production mystery (Del Trotter, Clark Bob parq rows
  existing post-Item-20 migration despite SQL-layer rollback in tests)
  was NOT fully chased. At SQL level, INSERT...RETURNING with permission
  denied DOES roll back the row (verified directly). At PostgREST HTTP
  level, the row must NOT roll back — that's the only way prod parq
  rows could exist. The behaviour difference between SQL and PostgREST
  on RETURNING failures is real but not yet investigated. Not worth
  chasing further since the .select() removal eliminates the issue
  entirely.
- The fix (one line in index.html, line ~1554): drop the `.select()` from
  the parq insert. The returned value was never consumed. Same data
  goes in, no RETURNING is requested, no 401 fires. Production now
  also runs without the silent console error.
- CB-33 strengthening (deferred from Session 19) landed cleanly: imports
  `getParqByCustomerId` from admin-db.js, asserts parq row exists with
  matching customer_id, booking_id, print_name, sign_date, and that
  q1_heart and q12_other_reasons are 'No'. All 7 tests in
  cb-01-new-client-happy-path.spec.js pass (33.9s wall clock).
- Pattern to watch for: any `.insert(...).select()` chain on a table
  where anon has INSERT but not SELECT will silently fail in production
  with the same symptom. Search for similar chains before adding new
  anon-write paths. Currently only parq is affected.

SESSION 24 LEARNINGS — TRIGGER OVERRIDES DIRECT UPDATE, SUCCESS-VIEW OVERLAY, BOOKINGS TABLE COLUMNS

  the block with REAL booking rows from the start (one INSERT per

  cause any subsequent bookings INSERT/DELETE on the same block. For

  hidden before the next user-facing action.
- The admin Bookings table (#btbody) renders 7 columns:
    Client (first + last, no email), Class, When, Block, Paid, Status, Actions
  Email is stored on the booking's customer record but is NOT shown in
  the row. Spec assertions that filter rows by email will silently

SESSION 25 LEARNINGS — DUPLICATE DETECTION LAYERS, TRANSIENT UI STATE, FIXTURE-PRESERVING CLEANUP
-------------------------------------------------------------------------------------------------
- Duplicate-booking protection has THREE independent layers that EC-08,
  EC-12, EC-13 each cover separately:
    (1) Front-end Step-1 early-detection — has_active_booking_on_block RPC
        called from goStep2() for RETURNING clients. Surfaces the friendly
        "You're already booked!" view BEFORE the user fills the rest of
        the form. Covered by CB-31.
    (2) RPC layer — book_if_available catches the partial unique index
        violation in PL/pgSQL and re-raises as RAISE EXCEPTION
        'ALREADY_BOOKED'. The front-end keys on this exact string at
        index.html line 1533. Covered by EC-13 (RPC behaviour) and
        EC-08 (front-end response when RPC is called directly from a
        race condition).
    (3) Partial unique index bookings_unique_active_per_block — fires on
        any direct INSERT (including outside the RPC). Last-line defence.
        Covered by EC-12.
  When any of these three regresses, a different test will catch it,
  but each layer needs its own test to avoid silent gaps.
- Transient UI state (EC-09's "Reserving..." button text) requires
  expect.poll with short intervals (50/100/200ms) and a small timeout
  (4s). The polled callback must snapshot both fields in a single
  evaluate() so they reflect the same moment — otherwise the disabled
  flag and text content can be read on different microtasks and one
  might already be reset. Pattern:
      await expect.poll(
        () => page.locator('#reserve-btn').evaluate(btn => ({
          disabled: btn.disabled,
          text:     (btn.textContent || '').trim()
        })),
        { timeout: 4000, intervals: [50, 100, 200] }
      ).toEqual({ disabled: true, text: 'Reserving...' });
- Fixture-preserving cleanup for "destructive setup" specs (EC-10): when
  a spec needs to delete real fixture bookings as test setup, the
  afterEach pattern is to snapshot the rows pre-delete and re-INSERT them
  with ON CONFLICT DO NOTHING in cleanup. Works because bookings rows
  have no auto-only fields that change on re-insert (the serial id is
  the only one and is not preserved). Combined with
  resyncBlockBookedCount this restores the fixture to a state
  indistinguishable from the seeded version. Use sparingly — entry-side
  pg INSERT of pre-existing rows is cheaper than full reseed but more
  expensive than the standard afterEach + deleteCustomerCascade pattern.
- pg error objects expose .code (SQLSTATE), .constraint (constraint
  name), and .column (column name for 23502 not-null violations). All
  three are stable identifiers worth asserting on — independent of
  Postgres locale or version-specific wording. EC-12 and EC-14 use all
  three; future DB-constraint specs should follow the same pattern.
- The Excel scenarios DO NOT track spec filenames in the Notes column —
  TEST-PLAN.md's Coverage Tracker is the source of truth for the
  scenario-to-spec mapping. When updating the Excel after a batch, only
  wording/SQL gets touched, never the Notes column unless the scenario

SESSION 30 LEARNINGS — AC SUITE, DATE TIMEZONE BUG, ACCORDION MECHANICS
------------------------------------------------------------------------
- Nav button for the public schedule is #nb-schedule, NOT #nb-home. Using
  #nb-home causes an immediate 30s timeout — the element doesn't exist.
- Classes and blocks inserted via direct pg before page.goto ARE loaded into
  the app's in-memory arrays on page init. Insert before goto; do not insert
  after. If a class is inserted after goto, the app won't know about it and
  validateAbDate() will silently fail to find the class for day-of-week check.
- toISOString() on a local midnight Date converts to UTC, shifting the date
  one day back in BST (UTC+1). For any date string passed into a browser date
  input where the app checks the day of week, always use toLocalISO():
    function toLocalISO(d) {
      return d.getFullYear() + '-'
        + String(d.getMonth() + 1).padStart(2, '0') + '-'
        + String(d.getDate()).padStart(2, '0');
    }
  This affects any test that calls validateAbDate() via the browser — i.e.
  any spec that fills #ab-start and triggers day validation. Direct pg inserts
  are unaffected (the DB doesn't validate day of week).
- toggleClassGroup() uses style.display directly ('block'/'none'), NOT a CSS
  class toggle. The .on class is defined in CSS but never added by the JS.
  Assert toBeVisible() (computed display) not toHaveClass(/on/).
- To expand a class group accordion reliably, target it by its onclick
  attribute: page.locator(`[onclick="toggleClassGroup('cg-${classId}')"]`).
  Filtering .class-group-header by hasText is unreliable — the click may
  land on a child element and fail to fire the handler.
- After saveNewBlock() fires renderDashboard(), the By Class accordion is
  rebuilt from scratch with all groups closed. Switching tabs (bookings →
  classes) re-renders the accordion but the in-memory blocks array state
  is not always reliable immediately post-save. Safest pattern for asserting
  By Class accordion contents after a UI block save: reload the full page
  so the app re-fetches from DB, then re-login, then expand the group.

SESSION 33 LEARNINGS — EMAIL NOTIFICATIONS SESSION 1, SUPABASE CLI, RESEND
--------------------------------------------------------------------------
- Resend does not offer a dedicated test API key type. All keys are live and
  deliver real emails. The test/prod distinction is handled by passing
  `isTest: true` in the Edge Function payload — the function redirects all
  recipients to `delivered@resend.dev` (Resend's silent sink). This is
  visible in the Resend dashboard logs but nothing lands in a real inbox.
- Supabase CLI requires Docker Desktop to bundle Edge Functions. Both must
  be installed and Docker must be running before `supabase functions deploy`.
- `supabase login` may fail silently if macOS Keychain blocks the token.
  Workaround: create a personal access token at
  https://supabase.com/dashboard/account/tokens and set it via:
    export SUPABASE_ACCESS_TOKEN=your_token_here
  This env var is session-scoped — must be re-set in each new terminal.
- `supabase link --project-ref <id>` only needs to be run once per project
  per machine. After that, `supabase functions deploy <name> --project-ref <id>`
  is the deploy command. The `--project-ref` flag must be supplied explicitly
  to target a specific project (prod vs test).
- Edge Function lives at `supabase/functions/send-email/index.ts`. This path
  is the Supabase CLI convention — do not move it.
- The `send-email` Edge Function payload: `{ to, subject, html, isTest }`.
  `isTest` must be explicitly passed from index.html — the function has no
  way to detect the environment itself.
- .insert().select() pitfall on settings table: existing pattern uses
  .upsert() with ON CONFLICT (key) DO UPDATE — this is correct. Never add
  a trailing .select() to anon writes on tables where anon lacks SELECT.

SESSION 34 LEARNINGS — EMAIL TRIGGER 1, NOEMAIL FLAG, TEST NOISE

- Every spec that completes a booking via the UI (CB-01, CB-03, CB-13 etc.)

  the fetch and assert the payload. The intercept prevents any real network
  call reaching Resend even on APP_PATH_EMAIL.
- buildReservedEmailHtml() uses table-based HTML (not CSS flexbox/grid) for

  font on bank details and border-left on the alert banner may render slightly
  differently, but the email is clean and readable in all major clients.
- The booking flow step structure for new clients:
    Step 1 → goStep2() button (no id — use button[onclick="goStep2()"])
    Step 2a (PAR-Q) → requires age, print name, AND #b-declaration checked

SESSION 35 LEARNINGS — EMAIL TRIGGER 2, EMAILPROMISE PATTERN
-------------------------------------------------------------
- confirmBookingAdmin() fires a "Confirming..." toast immediately (synchronous),
  then the async function updates the DB and calls sendBookingEmail(). A spec
  that waits for `#toastEl.on` will resolve on the FIRST toast before the email
  call has run, leaving capturedPayload null. Fix: use a Promise that resolves
  inside the page.route() callback (emailPromise pattern). The test awaits
  emailPromise, which only resolves when the intercept fires — guaranteeing
  the payload is captured before assertions run.
- The emailPromise pattern is now the canonical approach for any spec that
  needs to assert an email call triggered by an async admin action. SE-13
  is the reference implementation.
- Transient CI seed failures (exit code 1 during npm run seed on GitHub
  Actions) can occur without any code change — a momentary connection
  hiccup between GitHub's servers and Supabase. Re-running the job resolves
  it. Not worth investigating unless it recurs consistently.

SESSION 36 LEARNINGS — EMAIL TRIGGER 5, ARRAY INTERCEPT PATTERN, DYNAMIC DASHBOARD URL
---------------------------------------------------------------------------------------
- When multiple Edge Function calls fire in the same booking flow (client email + admin
  alert), a single `capturedPayload` variable in page.route() will be overwritten by the
  last call. Always collect into an array when more than one call is expected:
    `const capturedPayloads = []; ... capturedPayloads.push(body);`
  Assert by index: index 0 = first call (client), index 1 = second call (admin).
  SE-12 was broken by the addition of trigger #5 and fixed with this pattern.
- Dashboard URL in admin alert emails is built dynamically:
    `window.location.origin + window.location.pathname + '#dashboard'`
  This works correctly on GitHub Pages, Netlify (`book.lg-pilates.co.uk`), or any
  other host — no hardcoded domain needed.
- The admin alert `if(appSettings.adminEmail)` guard means no send attempt if Louise's
  email isn't configured in the settings table. Safe default.
- For SE-14 test 2 (returning client), using the fixture customer `returning-one@test.example`
  means `lookup_customer` finds them and skips PAR-Q. Cleanup must use
  `deleteBookingsForCustomerOnBlock` (not `deleteCustomerCascade`) to preserve the fixture
  customer row. Track the block ID via `getBlockByRole('fri-upcoming')` in the cleanup.
- GitHub Actions Node.js 20 deprecation warning: `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true`
  must be set as a top-level `env:` in the workflow file, not inside a step's env block,
  for it to apply to the action runners themselves. Fixed in `.github/workflows/playwright.yml`.
