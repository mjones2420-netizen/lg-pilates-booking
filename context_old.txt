DATABASE TABLES
---------------
classes                  — permanent class slots (id, name, level, day, time, end_time, venue, loc)
blocks                   — each 6-week block run (id, class_id, start_date, end_date, weeks, dates[], price, cap, booked, visible, status)
bookings                 — client bookings (id, class_id, block_id, customer_id, status, amount_due, refund_amount, created_at)
customers                — client records (id, first_name, last_name, email, phone, customer_type, priority, created_at)
                           NOTE: customers.priority column is retained in DB but no longer used by the app.
                           Per-class manual priority is now managed via customer_class_priority table.
parq                     — health forms (id, booking_id, customer_id, age, emergency_name, emergency_relationship, emergency_phone, q1_heart...q12_other_reasons, yes_details, additional_notes, print_name, sign_date)
                           NOTE: sign_date is DATE type.
                           Front-end stores ISO (YYYY-MM-DD) in a data-iso attribute on the display
                           element and sends that direct to the DB.
                           Admin PAR-Q view uses formatDateDisplay() helper to render "18 Apr 2026".
settings                 — key/value store for bank details (bank_name, bank_sort_code, bank_account_no)
waitlist                 — placeholder for future waitlist feature (id, class_id, customer_id, created_at)
                           RLS enabled 26 Mar 2026. Empty — not yet connected to booking flow.
cancellations            — audit trail of all client removals from blocks
                           Columns: id, customer_id, class_id, block_id, first_name, last_name, email,
                           class_name, venue, block_start_date, block_end_date, sessions_attended,
                           sessions_remaining, price_per_session, refund_amount, refunded (BOOLEAN),
                           refunded_at (TIMESTAMPTZ), cancelled_at (TIMESTAMPTZ)
                           RLS: admin_all_cancellations — authenticated only | ALL
                           NOTE: first_name/last_name/email/class_name/venue are denormalised so records
                           survive even if the customer or class is later deleted.
customer_class_priority  — per-class manual priority grants
                           Columns: id (BIGSERIAL), customer_id (BIGINT FK → customers), class_id (BIGINT FK → classes), granted_at (TIMESTAMPTZ)
                           UNIQUE constraint on (customer_id, class_id)
                           RLS: admin_all_customer_class_priority — authenticated only | ALL
                           ON DELETE CASCADE on both FKs — rows auto-delete if customer or class is deleted
                           This table replaces the global customers.priority boolean for manual priority.
                           A row present = that customer has manual priority for that class.

NOTE: parq table uses emergency_name, emergency_relationship, emergency_phone (NOT emergency_contact — that column was dropped)
NOTE: parq.booking_id FK is ON DELETE CASCADE — PAR-Qs auto-delete when their booking is deleted.
NOTE: bookings.block_id FK is ON DELETE CASCADE — prevents orphan bookings.
NOTE: App-level deletion code still clears child rows explicitly as a belt-and-braces safety net.
NOTE: When bulk-deleting all client data, deletion order must be:
parq → cancellations → bookings → customer_class_priority → customers

DATABASE CHECK CONSTRAINTS
---------------------------
Enforced at DB level for defence in depth and data integrity:

bookings:
  - bookings_status_check: status IN ('reserved','confirmed','cancelled','refund-pending','refunded','waitlist')

blocks:
  - blocks_status_check:    status IN ('upcoming','active','completed','cancelled')
  - blocks_price_nonneg:    price >= 0
  - blocks_cap_positive:    cap >= 1
  - blocks_weeks_positive:  weeks >= 1

customers:
  - customers_type_check:      customer_type IN ('new','returning','vip') OR customer_type IS NULL
  - customers_first_name_len:  char_length(first_name) <= 100
  - customers_last_name_len:   char_length(last_name)  <= 100
  - customers_email_len:       char_length(email)      <= 200
  - customers_phone_len:       char_length(phone)      <= 40

parq:
  - parq_yes_details_len:      char_length(yes_details)            <= 4000
  - parq_additional_notes_len: char_length(additional_notes)       <= 4000
  - parq_emergency_name_len:   char_length(emergency_name)         <= 100
  - parq_emergency_rel_len:    char_length(emergency_relationship) <= 50
  - parq_emergency_phone_len:  char_length(emergency_phone)        <= 40
  - parq_print_name_len:       char_length(print_name)             <= 100

IMPORTANT:
customer_type must be lowercase 'new' or 'returning' — the customers_type_check
constraint rejects capitalised variants.

NOT NULL CONSTRAINTS
--------------------
customers:
  - email         NOT NULL
bookings:
  - customer_id   NOT NULL
  - class_id      NOT NULL
  - block_id      NOT NULL
blocks:
  - class_id      NOT NULL
  - start_date    NOT NULL
  - end_date      NOT NULL
classes:
  - name          NOT NULL
  - day           NOT NULL
  - venue         NOT NULL

A NOT NULL violation returns Postgres error code 23502; existing try/catch
blocks surface a generic toast ("Error saving class." / "Error adding block.").

PARTIAL UNIQUE INDEX
--------------------
bookings_unique_active_per_block:
  CREATE UNIQUE INDEX ON bookings (customer_id, block_id) WHERE status != 'cancelled'

Prevents duplicate non-cancelled bookings by same customer on same block, at DB level.
Re-booking after cancellation still works — cancelled rows excluded by the WHERE clause.
book_if_available RPC catches unique_violation errors from this index and re-raises
as ALREADY_BOOKED so the front-end handles it identically.

SECURITY DEFINER FUNCTIONS (in Supabase)
-----------------------------------------
These bypass RLS and are called from the JS client instead of direct table access:
- lookup_customer(p_email) — returns customer record by email
- upsert_customer(p_first_name, p_last_name, p_email, p_phone, p_customer_type) — inserts or updates customer, returns id
- book_if_available(p_block_id, p_class_id, p_customer_id, p_amount_due) — atomic capacity check + booking insert, returns booking id
- check_priority_access(p_email, p_block_id) — checks if customer has priority for a block,
  returns BOOLEAN only. Checks customer_class_priority (per-class manual) then previous-block
  confirmed booking.
- has_active_booking_on_block(p_customer_id, p_block_id) — returns BOOLEAN only — true if customer has
  a non-cancelled booking on this block. Used by the booking modal step-1 handler.
- get_schema_fingerprint() — returns (fingerprint TEXT, total_objects BIGINT). Read-only deterministic
  hash of the public schema's structure: columns, constraints, indexes, RLS policies, functions, triggers,
  grants. Self-excluded from its own fingerprint. Used by the schema-check helper to verify production
  and test schemas remain identical (Session 12). Reveals nothing sensitive — just a hash and a count.

All functions granted EXECUTE to anon and authenticated roles.

DATABASE TRIGGERS
-----------------
- trg_sync_block_booked_count — fires AFTER INSERT OR DELETE on bookings table.
  Calls sync_block_booked_count() which recalculates blocks.booked from actual bookings
  (excluding cancelled status). Uses BIGINT for block_id.
  SECURITY DEFINER, SET search_path = public.

  IMPORTANT: This trigger fires on app-level inserts/deletes only. If bookings are
  bulk-deleted via raw SQL (e.g. in Supabase SQL editor), the booked count will NOT update
  automatically. After any raw SQL deletion of bookings, run this resync query:

    UPDATE blocks b
    SET booked = (
      SELECT COUNT(*) FROM bookings
      WHERE block_id = b.id
      AND status != 'cancelled'
    );

  Same applies to raw SQL INSERTs — resync after bulk seeding.

RLS POLICIES
------------
RLS enabled on all tables. Policy summary:

classes:                SELECT anon+auth | INS/UPD/DEL auth only
blocks:                 SELECT anon+auth | INS/UPD/DEL auth only
bookings:               INSERT anon+auth (routes via SECURITY DEFINER) | SELECT/UPD/DEL auth only
customers:              INSERT anon+auth (routes via SECURITY DEFINER) | SELECT/UPD/DEL auth only
parq:                   INSERT anon+auth | SELECT/DEL auth only (no UPDATE — write-once)
settings:               SELECT anon+auth | INS/UPD auth only
waitlist:               ALL auth only
cancellations:          ALL auth only
customer_class_priority: ALL auth only

NOTE: customers and bookings have no anon SELECT policy — intentional. All public reads
of customer/booking data must go via SECURITY DEFINER functions (lookup_customer,
check_priority_access, has_active_booking_on_block).

ANON ROLE GRANTS
----------------
Table grants to anon:
  classes   — SELECT only
  blocks    — SELECT only
  settings  — SELECT only
  parq      — INSERT only (PAR-Q save during booking)

Tables with NO anon grants at all:
  bookings, customers, cancellations, waitlist, customer_class_priority

Function EXECUTE grants to anon:
  book_if_available, check_priority_access, has_active_booking_on_block,
  lookup_customer, upsert_customer, get_schema_fingerprint

The `authenticated` role retains `ALL` on every table (admin dashboard unaffected).

---------------------------------------
The test database (ngzfhamjuviwfwuncrjo) is a structural mirror of production
(mrlooyixnlxzcfmvnqme). Parity is now verifiable on demand via the
schema-check helper.

How it works:
- Both projects have a SECURITY DEFINER function get_schema_fingerprint() that
  returns a deterministic MD5 hash + object count covering: columns,
  constraints, indexes, RLS policies, functions, triggers, and grants in the
  public schema. Read-only, granted to anon, exposes nothing sensitive.
- scripts/schema-check.js calls the RPC on both projects via their anon keys
  and compares the results.
- Auto-generated constraint and index names are normalised so cosmetic
  differences don't trigger a false positive.

    npm run schema-check
